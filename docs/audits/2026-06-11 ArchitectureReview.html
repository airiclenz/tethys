Corrections to the audit

C-01 — “the core process does not start”: wrong in its main effect

asyncio.create_task(handleCoreActivities()) evaluates the call first. handleCoreActivities is a synchronous infinite while True — it never returns, so create_task is never reached and the claimed TypeError never fires. 

tethys_core.py:39  def handleCoreActivities():          # sync, never returns
tethys_core.py:41      while True:
tethys_core.py:42          radioWrapper.handleRadioEvents(timeOutInSec = 30)
tethys_core.py:43          actionEngine.handleActions(radioWrapper)
tethys_core.py:49  task_core = asyncio.create_task(handleCoreActivities())   # blocks INSIDE the argument
tethys_core.py:50  task_fan  = asyncio.create_task(fan.control_fan())        # never reached
  • Watering and radio DO run — synchronously, inside main(), before asyncio ever schedules anything.
  • What is broken: fan.control_fan() (line 50) is never scheduled — no PID fan control, the Pi can overheat. The asyncio scaffolding is vestigial.
  • Latent crash: if the loop ever raises and returns, then the TypeError path exists. tethys-core.service has Restart=always; the watchdog only restarts daily at 01:00.
  • The audit’s fix (async def + asyncio.to_thread for blocking I/O) is still the right one; the recommended action order changes — this is no longer “nothing else matters until it boots.” Pump safety and API auth move to the top.

loadSilentSchedules type filter — confirmed bug, but currently latent

The missing scheduleType filter (common.py:192-198) is real. But models.py:144-149 creates exactly one ScheduleType — "silent" — and the web UI (schedule.ts:36-44) hardcodes scheduleType: "silent". Every schedule that can exist today is silent, so the filter’s absence has no current effect. Downgrade CRIT→latent; fix it when the evaluator is deepened (candidate 2) so the first non-silent type doesn’t detonate it.

SQLite contention — corrected: api and web do not share a database

The audit asked to verify whether the two BASE_DIRs point at one file. They don’t: api/tethys_api/db.sqlite3 and web/tethys_web/db.sqlite3 are separate. The no-timeout / no-WAL concern stands only for the api db (3 gunicorn workers + the core posting through the API). The web db holds no app data — a deletion-test candidate of its own.

Deepening candidates

The verified findings are not 24 independent bugs — they cluster around a handful of shallow modules and missing seams. Each candidate below deepens one of them; fixing the cluster member-by-member without the deepening leaves the next bug in the same place.

Top recommendation

One pump-control module owns the GPIO

With C-01 corrected, the worst verified risk is a pump that turns on and never turns off — reachable three ways (unbounded duration, swallowed off-write error, manual activate with no auto-off). All three are the same missing module. Deepening it retires both critical findings, two mediums, and gives the repo its first test seam (lgpio adapter + fake = two adapters, real seam). Candidates 3 → 2 follow naturally; the API-auth hardening can land in parallel.
